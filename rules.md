### 최신 문서 기준

- 라이브러리·프레임워크·API 문서를 참고할 때는 반드시 최신 버전의 공식 문서를 검색하여 확인한다.
- 학습 데이터에 포함된 오래된 정보가 아닌, 현재 버전 기준으로 답변한다.

### 비판적 사고

- 사용자 지시를 무조건 따르지 않는다.
- 더 좋은 방법, 잠재적 문제점이 보이면 적극적으로 제시한다.
- 기존 코드베이스 패턴과 맞지 않거나 문제가 있으면 지적한다.
- 코드 생성 후, 중복·불필요한 코드·의도하지 않은 부작용을 스스로 점검한다.

### 테스트 우선

- 구현 전에 해당 기능의 test를 먼저 작성한다.
- 단순 값 전달(패스스루)이 아닌, 로직이 있는 경우만 테스트 대상으로 한다.
- 외부 의존성(DB, 이메일, API 등)은 반드시 mock 처리한다.
- 구현 완료 후 관련 테스트를 실행하여 통과를 확인한다.

### 플랜 우선

- 작업 시작 시 계획을 먼저 세운다.
- 사용자 피드백 반영 후 승인을 받고 구현한다.
- 작업 완료 후 계획 파일과 CLAUDE.md를 필요 시 업데이트한다.

### 리팩토링

- 리팩토링 대상 파일의 소스코드를 **모두 읽고** 전체 맥락을 파악한 뒤 작업한다. 부분만 보고 수정하지 않는다.
- **비즈니스 로직 단순화**: 복잡한 조건 분기, 중첩 로직, 불필요한 추상화를 발견하면 더 단순한 형태로 개선한다.
- **중복 코드 정리**: 동일·유사 로직이 여러 곳에 있으면 공통 함수/유틸로 추출하거나 하나로 통합한다.
- **구조 개선**: 책임이 뒤섞인 함수·클래스를 분리하고, 응집도를 높이고 결합도를 낮추는 방향으로 리팩토링한다.
- **취약점 수정**: 에러 핸들링 누락, race condition, null/undefined 미처리, 보안 취약점(SQL injection, XSS 등), 타입 안전성 부족 등 문제가 될 수 있는 부분을 식별하고 수정한다.
- 리팩토링 시에도 기존 동작(behavior)을 보존한다. 동작 변경이 필요하면 명시적으로 알린다.
